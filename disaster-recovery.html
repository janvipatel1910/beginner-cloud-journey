What is Disaster Recovery?

Disaster Recovery (DR) is the process of restoring systems and data after a catastrophic failure — hardware failure, natural disaster, human error or cyber attack. AWS enables DR strategies ranging from simple backups to fully active multi-region deployments.

Key DR Metrics: RTO and RPO

RPO — Recovery Point Objective

How much data can you afford to lose?
RPO defines the maximum acceptable age of data that must be recovered after a failure. It determines how frequently you need to back up data.

Example: RPO of 1 hour means you can afford to lose at most 1 hour of data.

RTO — Recovery Time Objective

How quickly must you restore service?
RTO defines the maximum acceptable downtime — the time from when a disaster occurs to when the service is fully restored.

Example: RTO of 4 hours means the service must be restored within 4 hours of a failure.

DR Strategies (Cheapest → Most Expensive)

1. Backup and Restore

  • Lowest cost, highest RTO/RPO
  • Back up data to S3 and restore from snapshots when needed
  • No running infrastructure in DR region — provision everything after failure
  • RTO: Hours; RPO: Hours
  • Use when: cost is priority and extended downtime is acceptable

2. Pilot Light

  • A minimal version of the environment runs in the DR region
  • Core services (database replicated via RDS Read Replica) are always on
  • On failure: scale up compute (EC2/ELB/Auto Scaling) around the pre-running core
  • RTO: Tens of minutes; RPO: Minutes
  • Analogy: a pilot light on a boiler — small flame always burning, can ignite quickly

3. Warm Standby

  • A scaled-down but fully functional version of the production environment runs in DR
  • All services running but at minimum capacity (e.g. smaller instance types)
  • On failure: scale up to full production capacity and redirect DNS
  • RTO: Minutes; RPO: Seconds to minutes
  • More expensive than Pilot Light but faster recovery

4. Multi-Site Active-Active

  • Full production environment running simultaneously in multiple regions
  • Traffic distributed across regions via Route 53 weighted/latency routing
  • On failure: Route 53 health checks redirect 100% of traffic to healthy region
  • RTO: Near zero; RPO: Near zero
  • Most expensive — you pay for full infrastructure in multiple regions
  • Required for mission-critical applications with zero downtime tolerance

Key AWS Services for DR

  • S3 + Cross-Region Replication — replicate data across regions
  • RDS Read Replicas (cross-region) — database replication for DR
  • Route 53 — failover routing and health checks for automatic DNS failover
  • Aurora Global Database — sub-second RPO, under 1 minute RTO for global DB failover
  • EC2 AMIs — copy AMIs across regions for quick instance launch in DR
  • AWS Backup — centralised backup management across multiple AWS services
  • CloudFormation — redeploy infrastructure from code in minutes
  • Elastic Disaster Recovery (DRS) — replicate on-premises or cloud servers to AWS

AWS Solutions Architect Exam Notes

  • RPO = how much data loss is acceptable; RTO = how long downtime is acceptable
  • Lower RTO/RPO = more expensive strategy
  • Backup & Restore = cheapest, longest RTO/RPO — just snapshots in S3
  • Pilot Light = core services running (DB), compute added on failover
  • Warm Standby = full stack running at reduced capacity, scale up on failover
  • Multi-Site Active-Active = full production in multiple regions, near-zero RTO/RPO, most expensive
  • Aurora Global Database = best managed DB option for near-zero RPO cross-region DR
  • Route 53 failover routing + health checks = automatic DNS-based DR
  • Use CloudFormation to redeploy infrastructure quickly in DR region
  • Exam often presents a scenario with RTO/RPO requirements and asks which DR strategy to use

Key Takeaways

  • RPO = data loss tolerance; RTO = downtime tolerance
  • Four DR strategies from cheapest to most expensive: Backup/Restore → Pilot Light → Warm Standby → Multi-Site Active-Active
  • Choose strategy based on RPO/RTO requirements and budget
  • Route 53 failover routing automates DNS-level disaster recovery
  • Aurora Global Database and S3 CRR are key services for data replication across regions
