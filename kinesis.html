🌊 Amazon Kinesis

Streaming Data Real-Time Processing AWS Solutions Architect Associate High Exam Importance
Overview Kinesis Services Core Concepts Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon Kinesis is a platform for collecting, processing and analysing real-time streaming data at massive scale. It enables you to get timely insights and react to new information as it arrives, rather than waiting for batch processing jobs to complete.

Kinesis Services

Kinesis Data Streams

Collects and stores streaming data for real-time processing. Data is retained for 1–365 days (default 24 hours). Consumers read from the stream and process data in real time.

  • Data divided into shards — each shard provides 1MB/s write and 2MB/s read
  • Multiple consumers can read the same stream simultaneously
  • Used when you need custom real-time processing with full control

Kinesis Data Firehose

The easiest way to load streaming data into data stores and analytics services. Fully managed — no administration required.

  • Delivers data to: S3, Redshift, OpenSearch, Splunk, HTTP endpoints
  • Can transform data using Lambda before delivery
  • Near real-time (buffering up to 60 seconds or 1MB)
  • No data retention — it delivers and moves on (not for replay)

Kinesis Data Analytics

Run SQL queries or Apache Flink applications on streaming data in real time. Read from Kinesis Data Streams or Firehose and write results to S3, Redshift, or other streams.

Kinesis Video Streams

Ingest, process and store video streams from millions of devices. Used for video playback, machine learning and analytics on video data.

Core Concepts

Shards (Data Streams)

A stream is divided into shards for parallel processing. Each shard can ingest 1MB/s or 1,000 records/s, and deliver 2MB/s. You scale throughput by increasing the number of shards.

Partition Key

Producers specify a partition key when writing records. Kinesis uses the partition key to determine which shard the record goes to. Records with the same partition key always go to the same shard, maintaining ordering within that key.

Retention Period

Data Streams retains records for 24 hours by default, up to 365 days. This allows consumers to replay data — useful for reprocessing after failures.

Security Considerations

  • Use IAM policies to control who can produce and consume from streams
  • Enable server-side encryption using AWS KMS for data at rest
  • Use VPC Endpoints to access Kinesis without internet traffic
  • Monitor stream metrics with CloudWatch — shard iterator age indicates consumer lag

Architecture Overview

Data Producers (IoT sensors / app logs / clickstreams)
        │
        ▼
Kinesis Data Streams
        │
        ├── Lambda (real-time processing)
        ├── EC2 Consumer Application
        └── Kinesis Data Analytics (SQL/Flink)

        OR

Kinesis Data Firehose
        │
        ├── S3 (data lake)
        ├── Redshift (data warehouse)
        └── OpenSearch (search & analytics)

Real-World Use Cases

  • Real-time application log processing and anomaly detection
  • IoT sensor data ingestion from millions of devices
  • Clickstream analytics for e-commerce websites
  • Financial fraud detection — analyse transactions in real time
  • Video streaming ingestion and processing

AWS Solutions Architect Exam Notes

  • Data Streams = real-time, custom processing, replay, multiple consumers, manage shards yourself
  • Firehose = managed delivery to S3/Redshift/OpenSearch, near real-time, no shard management
  • Kinesis vs SQS: Kinesis for real-time ordered streaming; SQS for decoupling and async processing
  • Kinesis Data Streams retains data for 24 hours default, up to 365 days
  • Firehose has no data retention — once delivered it's gone
  • Scale Data Streams by adding shards — each shard = 1MB/s in, 2MB/s out
  • Hot shard problem — poor partition key choice causes uneven load; use high-cardinality keys
  • Lambda can be triggered directly by Kinesis Data Streams records

Key Takeaways

  • Kinesis Data Streams — real-time ingestion with replay, requires shard management
  • Kinesis Firehose — managed delivery to S3/Redshift/OpenSearch, near real-time, zero admin
  • Use Kinesis when you need to process high-volume streaming data in real time
  • Choose partition keys carefully to distribute load evenly across shards
  • Kinesis Analytics lets you run SQL queries on live streams without writing code
