⚡ Amazon DynamoDB

NoSQL Database Serverless AWS Solutions Architect Associate High Exam Importance
Overview Features Core Concepts Capacity Modes Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon DynamoDB is a fully managed, serverless NoSQL database service that provides fast and predictable performance with seamless scalability. It handles trillions of requests per day and supports single-digit millisecond response times at any scale.

DynamoDB automatically manages the underlying infrastructure — you never need to provision servers, manage patches or worry about capacity planning for storage.

Key Features

  • Single-digit millisecond performance at any scale
  • Fully serverless — no servers to manage or provision
  • Automatic multi-region replication with Global Tables
  • DynamoDB Streams — capture item-level changes for event-driven processing
  • DynamoDB Accelerator (DAX) — in-memory cache for microsecond read performance
  • Point-in-time recovery (PITR) for up to 35 days
  • TTL (Time to Live) — automatically delete expired items to save cost
  • Transactions — ACID-compliant transactions across multiple items and tables

Core Concepts

Tables, Items and Attributes

DynamoDB organizes data in tables. Each table contains items (equivalent to rows in SQL). Each item has attributes (equivalent to columns), but items in the same table do not need to have the same attributes — DynamoDB is schemaless.

Primary Key

Every item must have a unique primary key. There are two types:

  • Partition Key only — a single attribute that uniquely identifies each item. DynamoDB uses the partition key's value to distribute data across partitions.
  • Partition Key + Sort Key (Composite) — two attributes together form the primary key. Multiple items can share the same partition key as long as the sort key differs. Enables range-based queries.

Secondary Indexes

  • Global Secondary Index (GSI) — an index with a different partition key and optional sort key from the base table. Allows queries on non-primary key attributes. Has its own read/write capacity.
  • Local Secondary Index (LSI) — same partition key as the base table but a different sort key. Must be created at table creation time.

DynamoDB Streams

Streams capture a time-ordered sequence of item-level modifications (inserts, updates, deletes). Commonly used to trigger Lambda functions for event-driven processing — e.g. sending notifications or syncing data.

DynamoDB Accelerator (DAX)

DAX is a fully managed, in-memory cache for DynamoDB that delivers up to 10x performance improvement — reducing response times from milliseconds to microseconds for read-heavy workloads. It is API-compatible with DynamoDB — no code changes required.

Global Tables

Global Tables provide multi-region, multi-active replication. Your application can read and write to any region and DynamoDB automatically replicates changes to all other regions — ideal for globally distributed applications with low-latency requirements.

Capacity Modes

Provisioned Capacity

You specify the number of reads and writes per second (Read Capacity Units and Write Capacity Units). Best when you can predict your traffic. Supports Auto Scaling to adjust capacity automatically.

On-Demand Capacity

DynamoDB automatically scales to accommodate any workload. You pay per request — no capacity planning needed. Best for unpredictable or variable traffic patterns.

Security Considerations

  • Use IAM policies to control access to tables and items at a fine-grained level
  • Enable encryption at rest using AWS KMS (enabled by default)
  • Use VPC Endpoints to access DynamoDB without traffic leaving AWS's network
  • Enable Point-in-Time Recovery (PITR) to protect against accidental deletes
  • Use TTL to automatically expire and delete sensitive or stale data

Architecture Overview

Application (Lambda / EC2)
        │
        ├── Reads (cache hit)
        │       ▼
        │     DAX (in-memory cache)
        │
        └── Reads / Writes
                ▼
        DynamoDB Table
                │
                ├── DynamoDB Streams → Lambda → Event Processing
                │
                └── Global Tables → Region B, Region C (multi-region replication)

Real-World Use Cases

  • Gaming leaderboards — real-time score updates at massive scale
  • Shopping cart and session storage — fast key-value lookups
  • IoT data ingestion — high-throughput writes from millions of devices
  • User profile management — flexible schema for varying user attributes
  • Real-time analytics dashboards — DynamoDB Streams + Lambda for live updates
  • Serverless backend APIs — Lambda + DynamoDB for fully serverless architecture

AWS Solutions Architect Exam Notes

  • DynamoDB is a serverless NoSQL key-value and document database
  • Primary key can be partition key only, or partition key + sort key (composite)
  • GSI allows queries on non-primary key attributes — has its own capacity
  • LSI must be created at table creation and uses the same partition key
  • DAX provides microsecond read performance — use for read-heavy workloads
  • DynamoDB Streams + Lambda is a common event-driven pattern in exam questions
  • Global Tables = multi-region, multi-active (read and write in any region)
  • Use On-Demand mode for unpredictable traffic; Provisioned for predictable workloads
  • DynamoDB is not suitable for complex relational queries or joins — use RDS for those

Key Takeaways

  • DynamoDB is a fully managed, serverless NoSQL database with single-digit millisecond performance
  • Design your data model around the primary key and access patterns from the start
  • Use DAX for read-heavy workloads requiring microsecond latency
  • Global Tables enable low-latency reads and writes from any AWS region
  • Use DynamoDB Streams with Lambda to build event-driven, reactive applications
