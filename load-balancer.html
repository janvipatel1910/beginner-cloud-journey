⚖️ Elastic Load Balancing (ELB)

Networking Service High Availability AWS Solutions Architect Associate High Exam Importance
Overview Types of ELB Core Concepts Security Architecture Use Cases Exam Notes Summary

Service Overview

Elastic Load Balancing (ELB) automatically distributes incoming application traffic across multiple targets — such as EC2 instances, containers, IP addresses, and Lambda functions — in one or more Availability Zones.

ELB improves the availability and fault tolerance of applications by ensuring no single instance bears all the traffic. If an instance becomes unhealthy, ELB stops routing traffic to it automatically.

Types of Load Balancer

Application Load Balancer (ALB)

Operates at Layer 7 (HTTP/HTTPS). Best for web applications, microservices and container-based architectures.

  • Routes traffic based on URL path, hostname, HTTP headers or query strings
  • Supports WebSockets and HTTP/2
  • Integrates with AWS WAF for security
  • Supports target groups: EC2 instances, Lambda functions, IP addresses, ECS tasks

Network Load Balancer (NLB)

Operates at Layer 4 (TCP/UDP/TLS). Best for ultra-high performance, real-time applications and static IP requirements.

  • Handles millions of requests per second with very low latency
  • Supports static and Elastic IP addresses per Availability Zone
  • Best for gaming, IoT, and real-time streaming applications

Gateway Load Balancer (GWLB)

Operates at Layer 3. Used for deploying, scaling and managing third-party virtual network appliances such as firewalls and intrusion detection systems.

Classic Load Balancer (CLB)

Legacy load balancer operating at Layer 4 and Layer 7. AWS recommends migrating to ALB or NLB for all new workloads.

Core Concepts

Target Groups

A target group routes requests to registered targets (EC2 instances, Lambda, IP addresses). Health checks are configured per target group — ELB only routes to healthy targets.

Listeners

A listener checks for incoming connection requests on a specified protocol and port (e.g. HTTP:80, HTTPS:443). Each load balancer requires at least one listener.

Health Checks

ELB regularly sends requests to targets to check if they are healthy. Unhealthy targets are automatically removed from the rotation until they pass health checks again.

Cross-Zone Load Balancing

When enabled, each load balancer node distributes traffic evenly across all registered targets in all enabled Availability Zones — not just within its own AZ.

Sticky Sessions

Sticky sessions (session affinity) route requests from a specific user to the same target for the duration of a session. Useful for stateful applications that store session data locally.

Security Considerations

  • Use HTTPS listeners and terminate SSL/TLS at the load balancer
  • Integrate ALB with AWS WAF to filter malicious traffic
  • Place load balancers in public subnets and backend instances in private subnets
  • Use Security Groups on ALB to control which traffic is allowed
  • Enable access logging to capture detailed request information

Architecture Overview

        │

        ├── AZ-B: EC2 Instance (healthy ✅)

        └── AZ-C: EC2 Instance (unhealthy ❌ — removed from rotation)

Real-World Use Cases

  • Distribute web traffic across multiple EC2 instances for high availability
  • Route API requests to different microservices using ALB path-based routing
  • Handle millions of TCP connections per second with NLB for gaming backends
  • Serve multiple domains from a single ALB using host-based routing rules
  • Integrate with Auto Scaling to automatically add/remove instances as load changes

AWS Solutions Architect Exam Notes

  • ALB = Layer 7, HTTP/HTTPS, path-based and host-based routing
  • NLB = Layer 4, TCP/UDP, extreme performance, static IP support
  • GWLB = Layer 3, used for network appliances (firewalls, IDS/IPS)
  • ELB performs health checks — unhealthy targets are automatically deregistered
  • ALB supports Lambda as a target — NLB does not
  • Cross-zone load balancing is enabled by default on ALB, disabled by default on NLB
  • Use sticky sessions for stateful apps — but prefer stateless architecture where possible
  • ELB is regional — it spans multiple AZs within a region
  • Always pair ELB with Auto Scaling for full elasticity and availability

Key Takeaways

  • ELB distributes traffic across multiple targets to improve availability and performance
  • Use ALB for HTTP/HTTPS web apps; NLB for high-performance TCP/UDP workloads
  • Target groups and health checks ensure traffic only reaches healthy instances
  • Place ELB in public subnets, backend instances in private subnets
  • Combine ELB with Auto Scaling for a fully elastic, highly available architecture
