🗄️ Amazon Relational Database Service (RDS)

Database Service Managed Service AWS Solutions Architect Associate High Exam Importance
Overview Features Database Engines Core Concepts Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon Relational Database Service (RDS) is a managed database service that makes it easy to set up, operate and scale a relational database in the cloud. RDS handles routine database tasks such as provisioning, patching, backup, recovery, failure detection and repair.

With RDS, you focus on your application and data — not on managing database infrastructure.

Key Features

  • Automated backups with point-in-time recovery (up to 35 days retention)
  • Multi-AZ deployments for high availability and automatic failover
  • Read Replicas to improve read performance and reduce load on primary
  • Automated software patching and maintenance windows
  • Storage Auto Scaling — automatically increases storage when needed
  • Encryption at rest and in transit
  • Monitoring via Amazon CloudWatch and Enhanced Monitoring

Supported Database Engines

  • Amazon Aurora — AWS's own MySQL/PostgreSQL-compatible engine, up to 5x faster than MySQL
  • MySQL — most popular open-source relational database
  • PostgreSQL — advanced open-source relational database
  • MariaDB — community-developed MySQL fork
  • Oracle — enterprise relational database
  • Microsoft SQL Server — enterprise relational database from Microsoft

Core Concepts

Multi-AZ Deployment

In a Multi-AZ deployment, RDS automatically provisions and maintains a synchronous standby replica in a different Availability Zone. If the primary instance fails, RDS automatically fails over to the standby with no manual intervention required. The standby is not used for read traffic — it exists solely for failover.

Read Replicas

Read Replicas use asynchronous replication to create read-only copies of your database. They can be used to offload read traffic from the primary database and improve performance. Read Replicas can be promoted to standalone databases if needed.

  • Up to 5 Read Replicas per primary instance (15 for Aurora)
  • Can be in the same region or a different region (cross-region Read Replica)
  • Read Replicas do not provide automatic failover — use Multi-AZ for that

Automated Backups

RDS automatically takes daily backups of your database and captures transaction logs throughout the day. This enables point-in-time recovery to any second within your retention period (1–35 days).

DB Snapshots

Manual snapshots are user-initiated backups stored in S3. Unlike automated backups, manual snapshots are retained until you explicitly delete them.

Amazon Aurora

Aurora is AWS's cloud-native relational database engine, compatible with MySQL and PostgreSQL. It features:

  • Up to 5x the throughput of standard MySQL, 3x PostgreSQL
  • Storage auto-scales in 10GB increments up to 128TB
  • Up to 15 Read Replicas with sub-10ms replica lag
  • Aurora Serverless — automatically scales compute capacity up and down
  • Multi-Master — multiple write nodes for highest write availability

Security Considerations

  • Always deploy RDS instances in private subnets within a VPC
  • Use Security Groups to allow only your application servers to connect
  • Enable encryption at rest using AWS KMS (must be enabled at creation)
  • Enable SSL/TLS encryption in transit for all database connections
  • Use IAM database authentication instead of password-based auth where possible
  • Enable deletion protection to prevent accidental database deletion

Architecture Overview

Application (EC2 / Lambda — private subnet)
        │
        ▼
RDS Primary Instance (private subnet — AZ-A)
        │
        ├── Synchronous replication
        │         ▼
        │   RDS Standby (Multi-AZ — AZ-B) ← automatic failover only
        │
        └── Asynchronous replication
                  ▼
            Read Replica (AZ-C) ← read traffic only

Real-World Use Cases

  • E-commerce platforms storing product catalogues, orders and customer data
  • Banking and financial applications requiring ACID-compliant transactions
  • Content management systems (WordPress, Drupal) backed by MySQL/Aurora
  • SaaS applications with multi-tenant relational data
  • Reporting databases using Read Replicas to offload heavy queries from production

AWS Solutions Architect Exam Notes

  • Multi-AZ = high availability and automatic failover (synchronous replication)
  • Read Replicas = performance and scalability (asynchronous replication) — NOT for failover
  • Multi-AZ standby cannot be used for reads — it is for failover only
  • Encryption must be enabled at creation — cannot encrypt an existing unencrypted instance directly
  • To encrypt an unencrypted DB: take a snapshot → copy snapshot with encryption → restore from encrypted snapshot
  • Aurora is up to 5x faster than MySQL and supports up to 15 Read Replicas
  • RDS is a managed service — you cannot SSH into the underlying instance
  • Automated backups are enabled by default and retained for 1–35 days
  • Manual snapshots persist indefinitely until deleted

Key Takeaways

  • RDS is a managed relational database — no server management required
  • Multi-AZ provides automatic failover across Availability Zones
  • Read Replicas improve read performance — not a substitute for Multi-AZ
  • Always deploy in private subnets with encryption enabled
  • Aurora is AWS's highest-performance relational engine — consider it for new workloads
