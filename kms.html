🔑 AWS Key Management Service (KMS)

Security Service Encryption AWS Solutions Architect Associate Very High Exam Importance
Overview Features Core Concepts Encryption Types Best Practices Architecture Use Cases Exam Notes Summary

Service Overview

AWS Key Management Service (KMS) is a managed service that makes it easy to create and control the cryptographic keys used to protect your data. KMS integrates with most AWS services to encrypt data at rest and in transit.

Key Features

  • Centrally manage encryption keys across AWS services and applications
  • Integrated with 100+ AWS services — S3, EBS, RDS, DynamoDB, Lambda and more
  • Automatic key rotation for AWS-managed keys (every year)
  • Full audit trail via AWS CloudTrail — every key use is logged
  • FIPS 140-2 Level 2 validated hardware security modules (HSMs)
  • Cross-account and cross-region key sharing
  • Key policies control who can use and manage each key

Core Concepts

KMS Keys (Customer Master Keys)

A KMS key is a logical representation of a cryptographic key. The key material never leaves KMS unencrypted. There are three types:

  • AWS Managed Keys — created and managed by AWS on your behalf for specific services (e.g. aws/s3, aws/ebs). Free, automatic rotation every year. You cannot manage or view the key policy.
  • Customer Managed Keys (CMK) — keys you create, manage and control. You define key policies, enable/disable rotation, and can delete them. $1/month per key.
  • AWS Owned Keys — owned and managed entirely by AWS, shared across multiple accounts. You have no visibility or control.

Data Keys

KMS uses envelope encryption: KMS generates a data key, which is used to encrypt your data locally. The data key itself is encrypted by the KMS key. Only the encrypted data key is stored alongside your data. To decrypt, KMS first decrypts the data key, then your application decrypts the data.

Key Policies

Key policies are the primary way to control access to KMS keys. Every KMS key must have a key policy. IAM policies alone are not sufficient — the key policy must also grant access.

Key Rotation

AWS Managed Keys rotate automatically every year. Customer Managed Keys support optional automatic rotation every year. When a key rotates, old key material is retained to decrypt data encrypted with it.

Encryption in AWS Services

S3 Encryption

  • SSE-S3 — S3 manages keys, AES-256. Enabled by default on all new buckets.
  • SSE-KMS — KMS manages keys. Provides audit trail, key rotation, and access control.
  • SSE-C — Customer provides the key. AWS does not store the key.
  • Client-Side Encryption — encrypt before uploading; AWS never sees plaintext.

EBS Encryption

EBS volumes can be encrypted using KMS keys. Encryption covers data at rest, data in transit between instance and volume, snapshots, and volumes created from snapshots. Encrypted snapshots can be shared cross-account.

RDS Encryption

Must be enabled at DB creation. Encrypts the underlying storage, automated backups, read replicas and snapshots.

Best Practices

  • Use Customer Managed Keys for sensitive data requiring audit, rotation control and cross-account access
  • Enable CloudTrail to log all KMS API calls — provides full audit history
  • Apply least privilege in key policies — separate key administrators from key users
  • Enable automatic key rotation for CMKs
  • Never delete a KMS key without ensuring all encrypted data has been re-encrypted or deleted — key deletion is irreversible after a 7–30 day waiting period

Architecture Overview

Application
    │
    ▼
KMS GenerateDataKey API
    │
    ├── Returns: Plaintext Data Key (use to encrypt data)
    └── Returns: Encrypted Data Key (store alongside data)

Application encrypts data with Plaintext Data Key
→ Discards Plaintext Data Key from memory
→ Stores Encrypted Data Key with ciphertext

To decrypt:
Application calls KMS Decrypt API with Encrypted Data Key
→ KMS returns Plaintext Data Key
→ Application decrypts data

Real-World Use Cases

  • Encrypt S3 buckets containing sensitive customer data
  • Encrypt RDS databases for compliance (HIPAA, PCI-DSS, GDPR)
  • Encrypt EBS volumes storing application data
  • Securely store and rotate application secrets
  • Enforce encryption policies across all AWS services from a central key store

AWS Solutions Architect Exam Notes

  • AWS Managed Keys — free, auto-rotated, but no control over key policy
  • Customer Managed Keys — $1/month, full control, support custom key policies
  • KMS uses envelope encryption — data encrypted with a data key, data key encrypted with KMS key
  • SSE-KMS provides an audit trail in CloudTrail — know when and by whom keys were used
  • KMS keys are regional — to share encrypted data cross-region, you must re-encrypt
  • Deleting a CMK has a waiting period of 7–30 days — cannot be cancelled after deletion
  • EBS encryption must be enabled at volume creation — cannot encrypt an existing unencrypted volume directly
  • To encrypt unencrypted EBS: create snapshot → copy snapshot with encryption → restore

Key Takeaways

  • KMS centralises encryption key management across all AWS services
  • Use Customer Managed Keys for compliance, auditing and cross-account use cases
  • Envelope encryption is the mechanism behind all KMS-based encryption in AWS
  • CloudTrail logs every KMS API call — essential for compliance audit trails
  • Never hard-code encryption keys — always use KMS or Secrets Manager
