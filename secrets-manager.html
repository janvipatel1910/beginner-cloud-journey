🔒 AWS Secrets Manager & Parameter Store

Security Service Credentials Management AWS Solutions Architect Associate High Exam Importance
Overview Secrets Manager Parameter Store Comparison Security Architecture Exam Notes Summary

Service Overview

AWS provides two services for securely storing and managing application secrets, configuration values and credentials: AWS Secrets Manager and AWS Systems Manager Parameter Store. Both prevent hard-coding sensitive values in application code or configuration files.

AWS Secrets Manager

Secrets Manager is purpose-built for storing, rotating and managing secrets such as database credentials, API keys and OAuth tokens.

Key Features

  • Automatic secret rotation — rotate credentials on a schedule without downtime
  • Native integration with RDS, Aurora, Redshift and DocumentDB for seamless rotation
  • Encrypted at rest using AWS KMS
  • Fine-grained access control via IAM and resource-based policies
  • Automatic replication across regions for multi-region applications
  • Audit access via CloudTrail
  • Cost: ~$0.40/secret/month + API call charges

How Secret Rotation Works

Application retrieves secret from Secrets Manager via API
        │
Secrets Manager checks rotation schedule
        │
Lambda rotation function called
        │
└── Creates new credential in database
└── Updates the secret in Secrets Manager
└── Old credential remains valid briefly (dual-user strategy)
└── Old credential deactivated after grace period

AWS Systems Manager Parameter Store

Parameter Store is a hierarchical key-value store for configuration data and secrets. It is free for standard parameters and lower cost than Secrets Manager for non-rotating values.

Parameter Types

  • String — plain text value
  • StringList — comma-separated list of values
  • SecureString — encrypted value using AWS KMS

Parameter Tiers

  • Standard — up to 4KB value, free, no advanced features
  • Advanced — up to 8KB, supports parameter policies (TTL/expiry), $0.05/parameter/month

Hierarchy

Parameters can be organized hierarchically using path-like names: /myapp/production/db-password, /myapp/staging/db-password. Applications can retrieve all parameters under a path with a single API call.

Secrets Manager vs Parameter Store

Feature                    Secrets Manager    Parameter Store
─────────────────────────────────────────────────────────────
Automatic Rotation         ✅ Yes             ❌ No (manual)
RDS Native Integration     ✅ Yes             ❌ No
Cost                       ~$0.40/secret/mo   Free (standard)
Max Value Size             64KB               4KB (standard)
Cross-Region Replication   ✅ Yes             ❌ No
Version History            ✅ Yes             ✅ Yes
KMS Encryption             ✅ Yes             ✅ Yes (SecureString)
Use Case                   DB creds, API keys Config values, flags

Security Considerations

  • Never store secrets in environment variables, code, or S3 buckets — use Secrets Manager or Parameter Store
  • Grant applications access via IAM roles — not IAM user credentials
  • Use resource policies on secrets to control cross-account access
  • Enable CloudTrail to audit every secret access
  • Set rotation schedules on database credentials — ideally every 30–90 days

Architecture Overview

Lambda / EC2 Application
        │
        ▼
AWS SDK → Secrets Manager GetSecretValue API
        │
        ▼
Returns decrypted secret (DB password, API key)
        │
        ▼
Application connects to RDS using retrieved credentials

Rotation Lambda (runs automatically on schedule)
        │
        ▼
Creates new DB user/password → Updates secret → Deactivates old

AWS Solutions Architect Exam Notes

  • Secrets Manager = automatic rotation, database credentials, API keys, higher cost
  • Parameter Store = configuration values, lower cost, no automatic rotation
  • When exam asks about automatic DB credential rotation → Secrets Manager
  • When exam asks about storing config flags or non-rotating values → Parameter Store
  • Both integrate with KMS for encryption at rest
  • Both support IAM for access control
  • Secrets Manager can replicate secrets across regions for multi-region apps
  • Lambda functions used for rotation must be in the same region as the secret

Key Takeaways

  • Never hard-code credentials — always use Secrets Manager or Parameter Store
  • Use Secrets Manager when you need automatic rotation — especially for RDS passwords
  • Use Parameter Store for application configuration and non-rotating values (cheaper)
  • Both encrypt values at rest using KMS and log access via CloudTrail
  • Grant access via IAM roles — never give applications IAM user credentials
