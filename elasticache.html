⚡ Amazon ElastiCache

In-Memory Cache High Performance AWS Solutions Architect Associate High Exam Importance
Overview Cache Engines Caching Patterns Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon ElastiCache is a fully managed in-memory caching service that supports Redis and Memcached. It improves application performance by allowing you to retrieve data from fast in-memory caches rather than from slower disk-based databases, reducing read latency from milliseconds to microseconds.

Cache Engines: Redis vs Memcached

Redis

  • Supports rich data structures: strings, lists, sets, sorted sets, hashes
  • Persistence — data can be saved to disk and restored after restart
  • Replication — supports read replicas for high availability
  • Multi-AZ with failover — automatic failover to replica
  • Supports pub/sub messaging, Lua scripting and transactions
  • Supports Sorted Sets — perfect for leaderboards and priority queues
  • Use Redis when you need persistence, replication, or complex data structures

Memcached

  • Simple key-value store — strings only
  • No persistence — data is lost on restart
  • No replication — no high availability built in
  • Multi-threaded — can utilise multiple CPU cores
  • Supports horizontal scaling via sharding across multiple nodes
  • Use Memcached for maximum simplicity and raw throughput

Caching Patterns

Lazy Loading (Cache-Aside)

The application checks the cache first. On a cache miss, it reads from the database and writes the result to the cache for future requests. 

Application → Check ElastiCache
    │
    ├── Cache HIT → Return data immediately ✅
    │
    └── Cache MISS → Read from RDS → Write to ElastiCache → Return data
  • Only requested data is cached — efficient memory use
  • Cache misses result in 3 round trips — initial latency on first request
  • Data can become stale — set TTL to expire old data

Write-Through

Every write to the database also writes to the cache. Cache is always up-to-date but may store data that is rarely read.

  • Data is never stale — always consistent
  • More write operations and memory usage — cache may store unused data

Session Store

Store user session data in ElastiCache so any application server can serve any user request — enables stateless EC2 instances behind a load balancer.

Security Considerations

  • Deploy ElastiCache in private subnets — never expose to the internet
  • Use Security Groups to allow access only from application servers
  • Enable Redis AUTH — password-based access control for Redis clusters
  • Enable encryption in transit (TLS) and encryption at rest (KMS) for Redis
  • Use subnet groups to control which VPC subnets the cluster is placed in

Architecture Overview

User Request
        │
        ▼
Application (EC2 / Lambda)
        │
        ├── ElastiCache (in-memory, microseconds)
        │         │
        │    Cache HIT → return immediately
        │
        └── RDS / DynamoDB (on cache miss, milliseconds)
                  │
              Write result to cache

Real-World Use Cases

  • Database query caching — reduce RDS read load for frequently accessed data
  • Session management — store user sessions across stateless web servers
  • Gaming leaderboards — Redis Sorted Sets for real-time rankings
  • Rate limiting — track API call counts per user per second
  • Real-time analytics — fast counters and aggregations

AWS Solutions Architect Exam Notes

  • Redis = persistence, replication, multi-AZ, rich data types, sorted sets
  • Memcached = simple, no persistence, no replication, multi-threaded
  • Use ElastiCache to reduce database read load — cache frequently accessed queries
  • Session store pattern = ElastiCache + stateless EC2 behind ALB — common exam scenario
  • Lazy loading = fetch on miss; Write-through = always write to cache on DB write
  • ElastiCache does not natively support SQL — it's a key-value store
  • For exam: if question mentions reducing RDS load or sub-millisecond reads → ElastiCache
  • For leaderboards or sorted rankings → Redis Sorted Sets

Key Takeaways

  • ElastiCache provides managed Redis and Memcached in-memory caching
  • Use Redis for persistence, replication, complex data types and high availability
  • Use Memcached for simple, high-throughput, non-persistent caching
  • Dramatically reduces database load and response times for read-heavy applications
  • Always deploy in private subnets and use Security Groups for access control
