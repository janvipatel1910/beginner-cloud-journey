📨 Amazon Simple Queue Service (SQS)

Messaging Service Decoupling AWS Solutions Architect Associate Very High Exam Importance
Overview Features Queue Types Core Concepts Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon Simple Queue Service (SQS) is a fully managed message queuing service that enables you to decouple and scale microservices, distributed systems and serverless applications.

SQS eliminates the complexity and overhead associated with managing and operating message-oriented middleware. Producers send messages to the queue; consumers poll the queue and process messages independently.

Key Features

  • Fully managed — no servers to provision or maintain
  • Unlimited throughput and unlimited number of messages in a queue
  • Messages retained for up to 14 days (default 4 days)
  • Supports messages up to 256KB in size
  • At-least-once delivery guaranteed
  • Dead Letter Queues (DLQ) for handling failed messages
  • Server-side encryption using AWS KMS
  • Long polling to reduce empty responses and cost

Queue Types

Standard Queue

  • Nearly unlimited throughput
  • At-least-once delivery — messages may be delivered more than once
  • Best-effort ordering — messages may arrive out of order
  • Best for maximum throughput where exact ordering is not required

FIFO Queue (First-In-First-Out)

  • Up to 3,000 messages per second with batching (300 without)
  • Exactly-once processing — duplicates are eliminated
  • Strict ordering — messages are delivered in the exact order they are sent
  • Best for financial transactions, order processing and any workflow requiring strict ordering
  • Queue name must end in .fifo

Core Concepts

Message Visibility Timeout

When a consumer reads a message, it becomes invisible to other consumers for the visibility timeout period (default 30 seconds, max 12 hours). If the consumer does not delete the message within this time, it becomes visible again and another consumer can process it.

Dead Letter Queue (DLQ)

A DLQ is a separate queue that receives messages that could not be successfully processed after a maximum number of receive attempts (MaxReceiveCount). Used for debugging and isolating failed messages without blocking the main queue.

Long Polling

Long polling waits up to 20 seconds for a message to arrive before returning an empty response. This reduces the number of empty API calls, lowering costs and reducing latency compared to short polling. Always prefer long polling in production.

Message Groups (FIFO)

In FIFO queues, messages with the same MessageGroupId are processed in order. Different group IDs allow parallel processing while maintaining order within each group.

Delay Queues

Delay queues postpone the delivery of new messages for a specified time (0–900 seconds). Messages are invisible to consumers for the delay period.

Security Considerations

  • Use IAM policies and SQS resource policies to control access
  • Enable SSE-SQS or SSE-KMS to encrypt messages at rest
  • Use VPC Endpoints to keep SQS traffic within the AWS network
  • Set appropriate visibility timeouts to prevent duplicate processing
  • Use DLQs to capture and analyse failed messages without data loss

Architecture Overview

Producer (EC2 / Lambda / Application)
        │
        ▼
SQS Queue (buffer / decoupler)
        │
        ▼
Consumer (EC2 / Lambda — polls the queue)
        │
        ├── Success → Delete message from queue
        │
        └── Failure (after MaxReceiveCount)
                    ▼
              Dead Letter Queue (DLQ)

SQS + Auto Scaling Pattern

SQS Queue (growing backlog)
        │
        ▼
CloudWatch Alarm (queue depth metric)
        │
        ▼
Auto Scaling Group (scale EC2 workers based on queue depth)

Real-World Use Cases

  • Order processing — decouple order placement from fulfilment system
  • Image/video processing — queue uploaded files for background processing
  • Email delivery — buffer email send requests to avoid overwhelming the mail service
  • Microservices communication — prevent one slow service from blocking others
  • Auto Scaling trigger — scale workers based on SQS queue depth

AWS Solutions Architect Exam Notes

  • Standard = at-least-once, best-effort order; FIFO = exactly-once, strict order
  • Visibility timeout — message hidden during processing; re-appears if not deleted in time
  • Dead Letter Queue — captures messages that fail processing repeatedly
  • Long polling (up to 20 seconds) reduces cost — always prefer over short polling
  • SQS is the go-to answer for decoupling components in architecture questions
  • SQS + Auto Scaling = scale workers dynamically based on queue depth
  • FIFO queue names must end in .fifo
  • Max message size is 256KB — use S3 + SQS for larger payloads (Extended Client Library)
  • Max retention period is 14 days
  • SQS does not push messages — consumers must poll the queue

Key Takeaways

  • SQS decouples producers from consumers — a core pattern for resilient architectures
  • Use Standard for maximum throughput; FIFO for strict ordering and exactly-once delivery
  • Always configure a Dead Letter Queue to capture failed messages
  • Use long polling to reduce cost and improve responsiveness
  • SQS + Lambda or SQS + Auto Scaling are the most common exam architecture patterns
