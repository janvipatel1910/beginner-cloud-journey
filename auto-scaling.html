📈 AWS Auto Scaling

Compute Service High Availability AWS Solutions Architect Associate High Exam Importance
Overview Features Core Concepts Scaling Policies Security Architecture Use Cases Exam Notes Summary

Service Overview

AWS Auto Scaling monitors your applications and automatically adjusts capacity to maintain steady, predictable performance at the lowest possible cost. It can scale EC2 instances, ECS tasks, DynamoDB tables, Aurora replicas and more.

Auto Scaling ensures you always have the right number of EC2 instances available to handle your application load — scaling out when demand increases and scaling in when demand decreases to save cost.

Key Features

  • Automatic scaling in and out based on demand or a schedule
  • Maintains application availability by replacing unhealthy instances
  • Integrates with Elastic Load Balancer for even traffic distribution
  • Supports multiple scaling policies for different scenarios
  • Spans multiple Availability Zones for high availability
  • Cost optimization — scale in during off-peak hours
  • Supports EC2, ECS, DynamoDB, Aurora and other AWS services

Core Concepts

Auto Scaling Group (ASG)

An Auto Scaling Group is a collection of EC2 instances that are managed together for scaling and availability purposes. You define:

  • Minimum capacity — the minimum number of instances always running
  • Maximum capacity — the upper limit on the number of instances
  • Desired capacity — the number of instances you want running at any given time

Launch Template / Launch Configuration

A launch template defines the configuration for the EC2 instances in the Auto Scaling Group — AMI, instance type, key pair, security groups, IAM role and user data scripts. Launch Templates are the modern replacement for Launch Configurations.

Scaling Cooldown

After a scaling activity, Auto Scaling waits for a cooldown period (default 300 seconds) before executing another scaling action. This prevents rapid fluctuations in capacity.

Health Checks

Auto Scaling performs health checks on instances. When an instance is marked unhealthy (by EC2 status checks or ELB health checks), Auto Scaling terminates it and launches a replacement automatically.

Scaling Policies

Target Tracking Scaling

Automatically adjusts capacity to maintain a target metric value. Example: keep average CPU utilization at 50%. This is the simplest and most recommended policy.

Step Scaling

Scales based on the size of an alarm breach. Different scaling adjustments apply depending on how far the metric exceeds the threshold. Example: add 2 instances if CPU > 60%, add 4 instances if CPU > 80%.

Simple Scaling

Scales based on a single CloudWatch alarm. After the scaling action, Auto Scaling waits for a cooldown period before evaluating the alarm again. Less flexible than Step or Target Tracking policies.

Scheduled Scaling

Scale capacity in advance of predictable load changes. Example: increase capacity every Monday at 8am and reduce it at 8pm. Useful for workloads with known traffic patterns.

Predictive Scaling

Uses machine learning to forecast future demand based on historical patterns and provisions EC2 capacity in advance. Ideal for cyclical workloads.

Security Considerations

  • Use IAM Roles in Launch Templates — never hard-code credentials
  • Place Auto Scaling instances in private subnets behind a load balancer
  • Use Security Groups to restrict inbound access to instances
  • Enable ELB health checks so ASG can replace instances the load balancer marks unhealthy
  • Use encrypted EBS volumes in Launch Templates for data security

Real-World Use Cases

  • E-commerce sites scaling up during seasonal peaks (Black Friday, Christmas)
  • Media streaming services scaling to handle live event traffic spikes
  • Development environments scaling down to zero outside business hours to save cost
  • Replacing unhealthy instances automatically without manual intervention
  • Deploying new application versions via rolling updates across the ASG

AWS Solutions Architect Exam Notes

  • Auto Scaling Groups have minimum, maximum and desired capacity settings
  • Target Tracking is the simplest and most recommended scaling policy
  • Scheduled scaling is for known, predictable traffic patterns
  • Predictive scaling uses ML to forecast demand — provisions capacity in advance
  • Use ELB health checks (not just EC2) so unhealthy instances are replaced before the ELB marks them down
  • ASG automatically replaces unhealthy instances — this is self-healing architecture
  • Auto Scaling spans multiple AZs — always deploy across at least 2 AZs
  • Cooldown period prevents rapid thrashing — default is 300 seconds
  • Launch Templates are preferred over Launch Configurations — they support versioning

Key Takeaways

  • Auto Scaling automatically adds or removes EC2 instances based on demand
  • Define minimum, maximum and desired capacity in the Auto Scaling Group
  • Use Target Tracking for simple metric-based scaling; Scheduled for predictable patterns
  • Always combine Auto Scaling with ELB for a fully elastic, highly available setup
  • Auto Scaling is a key component of cost optimization — only pay for what you need
