Service Overview

AWS Lambda is a serverless compute service that lets you run code without provisioning or managing servers. You upload your code and Lambda handles everything required to run and scale it with high availability.

With Lambda, you pay only for the compute time you consume — there is no charge when your code is not running. Lambda automatically scales from a few requests per day to thousands per second.

Key Features

  • No servers to manage — fully serverless
  • Automatic scaling — from 1 to thousands of concurrent executions
  • Pay per use — billed per request and per millisecond of execution time
  • Supports multiple runtimes — Python, Node.js, Java, Go, Ruby, .NET, and custom runtimes
  • Up to 15 minutes execution time per invocation
  • Up to 10GB memory (CPU scales proportionally)
  • Integrates natively with most AWS services
  • Lambda Layers — share common libraries and dependencies across functions
  • Lambda@Edge — run Lambda functions at CloudFront edge locations globally

Core Concepts

Function

A Lambda function is the unit of deployment — it contains your code and configuration (runtime, memory, timeout, environment variables, IAM role). Each function is stateless — it does not retain data between invocations.

Handler

The handler is the entry point method in your code that Lambda calls when the function is invoked. It receives an event object and a context object.

Event Object

The event contains the data passed to the function by the invoking service (e.g. an S3 upload notification, an API Gateway request, or a DynamoDB Stream record).

Execution Environment

Lambda creates an execution environment with your code and dependencies. For repeated invocations, Lambda may reuse the same environment (warm start) — objects declared outside the handler persist between invocations in the same environment, which can improve performance.

Cold Start

A cold start occurs when Lambda creates a new execution environment for the first time or after a period of inactivity. This adds latency (typically 100ms–1s). Provisioned Concurrency can eliminate cold starts for latency-sensitive applications.

Concurrency

Lambda scales by running multiple instances of your function simultaneously. The default concurrency limit is 1,000 per region (can be increased). You can set reserved concurrency to guarantee capacity for a function or limit its maximum concurrency.

Layers

Lambda Layers let you manage and share common dependencies (libraries, frameworks, custom runtimes) separately from your function code. A function can use up to 5 layers.

Event Triggers

Lambda can be triggered by many AWS services:

  • API Gateway — build serverless REST/HTTP APIs
  • S3 — trigger on object uploads, deletes or modifications
  • DynamoDB Streams — process database changes in real time
  • SNS / SQS — process messages from queues and notification topics
  • CloudWatch Events / EventBridge — run on a schedule (cron) or in response to AWS events
  • Kinesis — process streaming data in real time
  • ALB — serve HTTP requests via Application Load Balancer
  • Cognito — run custom authentication flows

Security Considerations

  • Assign an IAM Execution Role to each Lambda function — grant only the permissions needed
  • Use environment variables for configuration — store secrets in AWS Secrets Manager or Parameter Store, not in plain text
  • Deploy Lambda functions inside a VPC when they need to access RDS, ElastiCache or other private resources
  • Use resource-based policies to control which services can invoke the function
  • Enable AWS X-Ray tracing to monitor function performance and debug issues

Real-World Use Cases

  • Serverless REST APIs — API Gateway + Lambda + DynamoDB
  • Image processing — resize/compress images automatically when uploaded to S3
  • Real-time data processing — process Kinesis or DynamoDB Streams records
  • Scheduled tasks — run cron jobs using EventBridge without managing servers
  • Chatbots and voice assistants — backend logic for Alexa Skills or Lex bots
  • ETL pipelines — transform and load data between AWS services

AWS Solutions Architect Exam Notes

  • Lambda is serverless — no servers to manage, automatic scaling
  • Maximum execution timeout is 15 minutes — not suitable for long-running processes
  • Lambda is stateless — do not rely on local storage between invocations
  • Cold starts add latency — use Provisioned Concurrency to eliminate them for critical functions
  • Lambda functions inside a VPC need a NAT Gateway to access the internet
  • Use Lambda@Edge to run code at CloudFront edge locations globally
  • API Gateway + Lambda + DynamoDB is the classic serverless architecture pattern
  • Lambda can be triggered synchronously (API Gateway, ALB) or asynchronously (S3, SNS)
  • Default concurrency limit is 1,000 per region — use reserved concurrency to isolate critical functions

Key Takeaways

  • Lambda lets you run code without managing any infrastructure
  • It is event-driven — triggered by S3, DynamoDB, API Gateway, SQS, and many more
  • Pay only for actual execution time — extremely cost-efficient for variable workloads
  • Assign least-privilege IAM Execution Roles and never hard-code credentials
  • Combine with API Gateway and DynamoDB for a fully serverless application stack
