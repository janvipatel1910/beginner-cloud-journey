🐳 Amazon ECS & AWS Fargate

Container Service Serverless Option AWS Solutions Architect Associate High Exam Importance
Overview ECS Fargate EKS Core Concepts Security Architecture Exam Notes Summary

Service Overview

AWS provides fully managed container orchestration services so you can run containerised applications without managing the underlying infrastructure. The two main services are Amazon ECS (Elastic Container Service) and Amazon EKS (Elastic Kubernetes Service).

Amazon ECS (Elastic Container Service)

ECS is AWS's own fully managed container orchestration service. It runs Docker containers on a cluster of EC2 instances or on Fargate (serverless). ECS is simpler than Kubernetes and tightly integrated with AWS services.

ECS Launch Types

  • EC2 Launch Type — you provision and manage the EC2 instances in your cluster. More control, lower cost at scale.
  • Fargate Launch Type — AWS manages the underlying infrastructure. You just define CPU and memory requirements. Serverless containers.

AWS Fargate

Fargate is a serverless compute engine for containers. You do not provision, manage or scale EC2 instances — you simply define your container requirements and Fargate runs them.

Key Fargate Features

  • No EC2 instances to manage — fully serverless
  • Pay only for the vCPU and memory your containers use
  • Works with both ECS and EKS
  • Each task runs in its own isolated compute environment
  • Ideal for intermittent workloads, batch jobs and microservices

Amazon EKS (Elastic Kubernetes Service)

EKS is a fully managed Kubernetes service. Use EKS when your team already uses Kubernetes, needs Kubernetes-specific features, or is running a multi-cloud/hybrid Kubernetes strategy.

  • AWS manages the Kubernetes control plane
  • Runs on EC2 or Fargate (serverless nodes)
  • Use when you need Kubernetes compatibility or have existing K8s workloads
  • More complex than ECS but more portable across clouds

ECR (Elastic Container Registry)

Amazon ECR is a fully managed Docker container registry for storing, managing and deploying container images. Integrates natively with ECS and EKS. Images are encrypted at rest and scanned for vulnerabilities by Inspector.

Core Concepts (ECS)

Task Definition

A task definition is a blueprint for your application — it defines the Docker image, CPU/memory, environment variables, port mappings, IAM role and networking mode. Think of it as a Dockerfile for your entire application stack.

Task

A running instance of a task definition. One task can run multiple containers.

Service

An ECS Service ensures a specified number of tasks are always running. If a task fails, the service replaces it automatically. Services integrate with ALB for load balancing and Auto Scaling for elasticity.

Cluster

A cluster is a logical grouping of tasks or services. For EC2 launch type, the cluster also groups the EC2 instances.

Security Considerations

  • Assign an IAM Task Role to each task — grants the container AWS service permissions
  • Use ECR image scanning (Inspector) to detect vulnerabilities before deployment
  • Run tasks in private subnets — expose only through ALB in public subnets
  • Use Secrets Manager or Parameter Store to inject secrets into containers as environment variables
  • Enable VPC Flow Logs to monitor container network traffic

Architecture Overview

Internet Traffic
        │
        ▼
Application Load Balancer (public subnet)
        │
        ▼
ECS Service (manages desired task count)
        │
        ├── ECS Task (Fargate) — Container A
        ├── ECS Task (Fargate) — Container B
        └── ECS Task (Fargate) — Container C

Each task:
  - Pulls image from ECR
  - Assumes IAM Task Role
  - Reads secrets from Secrets Manager
  - Stores data in RDS / DynamoDB / S3

AWS Solutions Architect Exam Notes

  • ECS = AWS-native container orchestration; simpler than Kubernetes
  • Fargate = serverless containers — no EC2 management; pay per task CPU/memory
  • EKS = managed Kubernetes — use when team already knows Kubernetes
  • ECR = managed Docker registry — stores and scans container images
  • ECS EC2 launch type = you manage instances; Fargate = AWS manages infrastructure
  • Use IAM Task Roles (not instance roles) to grant containers AWS permissions
  • ECS integrates with ALB for load balancing and Auto Scaling for elasticity
  • Fargate is best for variable or unpredictable workloads — no idle EC2 costs
  • For batch processing, use ECS Fargate or AWS Batch

Key Takeaways

  • ECS manages Docker containers on AWS — choose EC2 or Fargate launch type
  • Fargate = serverless containers, no server management, pay per use
  • EKS = managed Kubernetes — use for teams with existing K8s expertise
  • Always use IAM Task Roles for containers, not hard-coded credentials
  • Store images in ECR and enable vulnerability scanning with Inspector
