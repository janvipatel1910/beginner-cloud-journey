🔔 Amazon Simple Notification Service (SNS)

Messaging Service Pub/Sub AWS Solutions Architect Associate Very High Exam Importance
Overview Features Core Concepts SNS + SQS Fanout Security Architecture Use Cases Exam Notes Summary

Service Overview

Amazon Simple Notification Service (SNS) is a fully managed pub/sub (publish/subscribe) messaging service for application-to-application (A2A) and application-to-person (A2P) communication.

Publishers send messages to topics. SNS immediately pushes those messages to all subscribed endpoints — unlike SQS where consumers must poll.

Key Features

  • Pub/Sub model — one message delivered to many subscribers simultaneously
  • Push-based delivery — SNS pushes messages to subscribers instantly
  • Supports multiple subscriber types: SQS, Lambda, HTTP/HTTPS, Email, SMS, Mobile Push
  • Message filtering — subscribers only receive messages matching their filter policy
  • Message fanout — broadcast one message to multiple SQS queues or Lambda functions
  • SNS FIFO topics — strict ordering and deduplication for critical workflows
  • Up to 12.5 million subscriptions per topic
  • Server-side encryption using AWS KMS

Core Concepts

Topics

A topic is a logical access point and communication channel. Publishers send messages to a topic. SNS delivers that message to every subscribed endpoint.

Subscriptions

A subscription links a topic to an endpoint. Supported protocols:

  • SQS — deliver to an SQS queue (most common for system-to-system)
  • Lambda — trigger a Lambda function
  • HTTP/HTTPS — send to a web endpoint
  • Email / Email-JSON — send to an email address
  • SMS — send a text message to a phone number
  • Mobile Push — push to iOS/Android via APNs/FCM

Message Filtering

Each subscription can have a filter policy — a JSON object that defines which messages the subscriber receives based on message attributes. This allows different consumers to receive only the messages relevant to them without creating separate topics.

SNS FIFO Topics

Similar to SQS FIFO queues, SNS FIFO topics guarantee strict message ordering and exactly-once delivery. They can only deliver to SQS FIFO queues. Topic name must end in .fifo.

SNS + SQS Fanout Pattern

The fanout pattern is one of the most important architecture patterns in the AWS exam. A single SNS message is delivered to multiple SQS queues simultaneously, allowing parallel processing by different services. 

S3 Event / Application
        │
        ▼
SNS Topic
        │
        ├── SQS Queue A → Lambda (send email confirmation)
        │
        ├── SQS Queue B → EC2 Worker (process order)
        │
        └── SQS Queue C → Lambda (update analytics dashboard)

Benefits: decoupled, fully parallel processing, no data loss if one consumer is slow or fails.

Security Considerations

  • Use IAM policies and SNS resource policies to control who can publish or subscribe
  • Enable SSE-SNS or SSE-KMS to encrypt messages at rest
  • Use HTTPS endpoints only for HTTP subscriptions
  • Use message filtering to prevent subscribers from receiving irrelevant data
  • Use VPC Endpoints to keep SNS traffic within the AWS network

Architecture Overview

Publisher (EC2 / Lambda / CloudWatch Alarm)
        │
        ▼
SNS Topic
        │
        ├── Email Subscription    → Alert to admin
        ├── SMS Subscription      → Text to on-call engineer
        ├── SQS Subscription      → Queue for processing
        └── Lambda Subscription   → Serverless processing

Real-World Use Cases

  • System alerts — CloudWatch alarm triggers SNS → sends SMS/email to operations team
  • Order notifications — new order triggers SNS fanout to fulfilment, billing and analytics queues
  • Mobile push notifications — send targeted alerts to iOS and Android apps
  • Event broadcasting — broadcast events to multiple downstream microservices simultaneously

AWS Solutions Architect Exam Notes

  • SNS is push-based; SQS is pull/poll-based — know the difference
  • Fanout pattern = SNS → multiple SQS queues — very common exam scenario
  • SNS delivers to: SQS, Lambda, HTTP/S, Email, SMS, Mobile Push
  • Message filtering lets subscribers receive only relevant messages
  • SNS does not persist messages — if delivery fails and no retry succeeds, message is lost (use SQS for durability)
  • SNS FIFO + SQS FIFO = strict ordering with fanout
  • CloudWatch alarms use SNS to send notifications — know this integration
  • SNS cannot directly deliver to Kinesis or DynamoDB

Key Takeaways

  • SNS is a pub/sub service — one message, many subscribers
  • Push-based: SNS delivers to subscribers; consumers don't poll
  • Use SNS + SQS fanout to fan a single event out to multiple parallel workflows
  • Use message filtering to route specific messages to specific subscribers
  • SNS alone doesn't guarantee durability — pair with SQS for reliable processing
