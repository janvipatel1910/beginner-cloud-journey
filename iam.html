🔐 AWS Identity and Access Management (IAM)

Security Global AWS Service Solutions Architect Associate High Exam Importance
Overview Core Concepts IAM Components Best Practices Architecture Use Cases Exam Notes Summary

Service Overview

AWS Identity and Access Management (IAM) is a global AWS service used to securely control access to AWS resources.

IAM enables organizations to define who can access AWS resources, which resources they can access, and what actions they are authorized to perform.

Core Concepts

Authentication

Authentication verifies the identity of a user, application, or service attempting to access AWS resources.

  • Username and Password
  • Access Keys (Access Key ID + Secret Access Key)
  • Multi-Factor Authentication (MFA)

Authorization

Authorization determines which AWS resources an authenticated identity can access and what actions can be performed. IAM uses policies to grant or deny permissions.

IAM Components

IAM Users

An IAM User represents an individual identity within an AWS account. Each user has their own credentials and can be assigned permissions directly or through group membership.

IAM Groups

IAM Groups provide a mechanism for managing permissions across multiple users. A user can belong to multiple groups. Groups cannot contain other groups.

IAM Roles

IAM Roles provide temporary security credentials and are assumed by users, applications or AWS services. Roles are the preferred way to grant AWS services (like EC2) access to other services (like S3).

IAM Policies

Policies are JSON documents that define permissions — what actions are allowed or denied on which resources. They are attached to users, groups or roles.

Security Best Practices

  • Apply the Principle of Least Privilege — grant only the permissions required
  • Enable Multi-Factor Authentication (MFA) on all accounts, especially root
  • Never use or share the Root Account for everyday tasks
  • Rotate credentials and access keys regularly
  • Use IAM Roles instead of long-term Access Keys where possible
  • Monitor IAM activity using AWS CloudTrail

Architecture Overview

AWS Account
│
├── IAM Users         → individual identities
│
├── IAM Groups        → collections of users with shared permissions
│
├── IAM Roles         → temporary credentials for services or cross-account access
│
└── IAM Policies      → JSON permission documents attached to the above

Real-World Use Cases

  • Developer access management — different permissions per environment (dev/staging/prod)
  • Granting EC2 instances access to S3 using IAM Roles (no hard-coded credentials)
  • Cross-account AWS access for centralized security teams
  • Read-only auditor access to review AWS resources
  • CI/CD pipeline credentials using service-specific IAM Roles

AWS Solutions Architect Exam Notes

  • IAM is a Global Service — not region-specific
  • Policies define permissions using JSON
  • Groups cannot contain other groups
  • IAM Roles provide temporary credentials — preferred for services and cross-account access
  • Follow the Principle of Least Privilege
  • Root account should be protected with MFA and rarely used
  • Access keys are for programmatic access — rotate regularly

Key Takeaways

  • IAM controls authentication and authorization for AWS resources
  • Users represent individual identities; Groups simplify permission management
  • Roles provide temporary access — the preferred approach for services
  • Policies are JSON documents that grant or deny permissions
  • Always apply Least Privilege and enable MFA
