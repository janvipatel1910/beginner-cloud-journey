📡 Amazon Route 53

Service Overview

Amazon Route 53 is a highly available and scalable cloud Domain Name System (DNS) web service. It translates human-readable domain names (like www.example.com) into the numeric IP addresses that computers use to connect to each other.

Route 53 is named after port 53, the port used for DNS traffic. It is a global AWS service — not tied to any specific region.

Key Features

  • Domain registration — purchase and manage domain names directly in AWS
  • DNS routing — route traffic to AWS resources or external endpoints
  • Health checks — monitor the health of endpoints and route traffic away from unhealthy ones
  • Multiple routing policies for different traffic management needs
  • 100% availability SLA — guaranteed uptime for DNS resolution
  • Integration with AWS services — ALB, CloudFront, S3, EC2 and more
  • Private hosted zones — DNS resolution within a VPC

Core Concepts

Hosted Zones

A hosted zone is a container for DNS records for a specific domain. There are two types:

  • Public Hosted Zone — routes traffic on the public internet
  • Private Hosted Zone — routes traffic within one or more VPCs

DNS Records

  • A Record — maps a domain name to an IPv4 address
  • AAAA Record — maps a domain name to an IPv6 address
  • CNAME Record — maps a domain name to another domain name (cannot be used for root domain)
  • Alias Record — AWS-specific record that maps a domain to an AWS resource (e.g. ALB, CloudFront). Can be used for root domain and is free of charge
  • MX Record — routes email to mail servers
  • TXT Record — stores text information, used for domain verification

TTL (Time to Live)

TTL defines how long a DNS resolver caches a record before requesting a fresh copy from Route 53. Lower TTL means faster propagation of DNS changes but more queries to Route 53.

Routing Policies

Simple Routing

Routes traffic to a single resource. If multiple values are specified, Route 53 returns a random one. Does not support health checks on individual records.

Weighted Routing

Routes a percentage of traffic to different resources based on assigned weights. Useful for A/B testing or gradually shifting traffic between deployments.

Latency-Based Routing

Routes traffic to the AWS Region that provides the lowest latency for the user. Ideal for globally distributed applications.

Failover Routing

Routes traffic to a primary resource when it is healthy, and automatically fails over to a secondary resource when the primary becomes unhealthy. Requires health checks.

Geolocation Routing

Routes traffic based on the geographic location of the user. Useful for serving region-specific content or complying with data residency requirements.

Geoproximity Routing

Routes traffic based on the geographic location of resources and users. You can shift traffic by adjusting a bias value. Requires Route 53 Traffic Flow.

Multi-Value Answer Routing

Returns multiple healthy IP addresses in response to DNS queries. Similar to Simple Routing but with health checks on each record. Not a substitute for a load balancer, but improves availability.

Security Considerations

  • Use DNSSEC to protect against DNS spoofing attacks
  • Use private hosted zones for internal AWS resources to keep DNS resolution off the public internet
  • Configure health checks to automatically remove unhealthy endpoints from DNS responses
  • Use IAM policies to restrict who can modify Route 53 hosted zones and records

Real-World Use Cases

  • Register and manage a custom domain for your website
  • Route traffic across multiple AWS regions using latency-based routing
  • Implement automatic failover from primary to DR site using health checks
  • A/B test new application versions by splitting traffic with weighted routing
  • Serve different content to users in different countries using geolocation routing

AWS Solutions Architect Exam Notes

  • Route 53 is a global service — not region-specific
  • Alias records are free and can point to the root domain — use over CNAME for AWS resources
  • CNAME records cannot be used for the root/apex domain (e.g. example.com)
  • Failover routing requires health checks — primary/secondary setup
  • Weighted routing is used for blue/green deployments and A/B testing
  • Latency routing routes to the region with the lowest network latency, not geographic distance
  • Geolocation routing is based on user location — used for compliance and localized content
  • Health checks can monitor HTTP, HTTPS, TCP endpoints and CloudWatch alarms

Key Takeaways

  • Route 53 is AWS's highly available DNS and domain registration service
  • Hosted zones hold DNS records — public for internet, private for VPCs
  • Use Alias records for AWS resources — they're free and support the root domain
  • Choose the right routing policy for your use case: failover for DR, weighted for A/B, latency for global apps
  • Health checks enable automatic failover and keep traffic away from unhealthy endpoints
